Негативні ключові слова: пропуск нерелевантних вакансій до виклику Groq API

Проблема/Можливість: Функція is_relevant() у check_new.py перевіряла лише позитивні ключові слова (looker, bi developer, bi engineer тощо). Через це вакансії на кшталт «Power BI Developer», «Senior BI Developer (MS SQL + Tableau)», «BI Engineer with Qlik Cloud» і «Microsoft Fabric» проходили фільтр і надсилались до Groq API — де отримували оцінку 1–2/10 і відкидались. Сьогодні вночі так витрачено 8 API-викликів (із яких один викликав обмеження запитів і 30-секундне очікування), а жодна вакансія додана не була.

Рішення: 1) У config.py додано два нових списки: NEGATIVE_TITLE_KEYWORDS — 16 ключових слів, що ідентифікують нерелевантні інструменти (power bi, powerbi, ms sql, microsoft sql, microsoft fabric, azure synapse, qlik, ssrs, ssis, ssas, tableau, microstrategy, cognos, sap bo, sap bw, spotfire); POSITIVE_OVERRIDE_KEYWORDS — слова, що скасовують негативний фільтр (looker, lookml, bigquery, snowflake, dbt) — щоб «Looker + Tableau Developer» все одно пройшов. 2) Функцію is_relevant() у check_new.py розширено: якщо заголовок вакансії містить хоча б одне негативне ключове слово і жодного слова-перевизначення — функція повертає False ще до будь-якого звернення до Groq.

Змінені файли: config.py, check_new.py

Перевірка: venv/bin/python3 -c "import app; print('OK')" та import check_new — без помилок. 12 тест-кейсів з реальних вакансій пройшли успішно: усі 7 нерелевантних вакансій з нічного запуску (Power BI, MS SQL, Tableau, Qlik, Microsoft Fabric) коректно відфільтровані; «Middle Data Analyst / BI Developer», «Senior Analytics Engineer Looker» та «Looker + Tableau Developer» залишились.

Кнопка «Перевірити зараз»: ручний запуск пошуку вакансій без очікування крону

Проблема/Можливість: Автоматична перевірка нових вакансій запускається лише за розкладом (кожні 2 години). Якщо хотілося перевірити вакансії одразу після додавання нового джерела або в будь-який момент — доводилося або чекати, або запускати check_new.py вручну в терміналі.

Рішення: 1) У app.py додано два нових ендпоінти: POST /api/check — запускає check_new.py у фоновому потоці (через threading.Thread + subprocess.run); ідемпотентний — якщо перевірка вже виконується, просто повертає "already running". GET /api/check/status — повертає {"running": true/false}. 2) В index.html додано кнопку «Перевірити зараз» у рядок дій зліва від «Позначити переглянутими». При натисканні: іконка оновлення починає обертатися (@keyframes spin), текст змінюється на «Перевіряємо...», кнопка стає неактивною. JS кожні 4 секунди опитує /api/check/status; коли running стає false — автоматично перезавантажує дані (load() + loadStatus()) і відновлює кнопку. Є запасний таймаут 5 хвилин на випадок зависання.

Змінені файли: app.py, index.html

Перевірка: venv/bin/python3 -c "import app; print('OK')" — без помилок. Ендпоінти /api/check та /api/check/status зареєстровані (підтверджено через app.url_map). Логіка захисту від паралельних запусків перевірена: повторний POST /api/check повертає "already running" поки фоновий потік не завершиться.

2026-04-14 03:10

Липкий рядок фільтрів: фільтри та пошук залишаються видимими під час прокрутки

Проблема/Можливість: Коли список вакансій довгий і користувач прокручує сторінку вниз, панель фільтрів (All / Fav / New / Hot / OK / Skip / Stats) та рядок пошуку зникали з екрану. Щоб відфільтрувати або знайти вакансії, потрібно було прокручувати назад нагору — незручно, особливо коли в списку є десятки карток з декількох джерел.

Рішення: Фільтри та панель сортування/пошуку обгорнуто у контейнер .sticky-controls з position: sticky; top: 0; z-index: 30. При прокручуванні сторінки понад 60px до контейнера додається клас .scrolled через scroll-listener (passive), що вмикає тінь box-shadow: 0 1px 0 border-subtle, 0 4px 20px rgba(0,0,0,.45) — чітко відокремлюючи липку панель від контенту знизу. На різних розмірах екрану (mobile/tablet/desktop/wide) відступи скориговані через margin/padding у медіа-запитах, щоб панель розтягувалась на повну ширину. Панель статистики (Stats) та рядок дій (Розгорнути всі, Експорт CSV тощо) залишені поза sticky-зоною, щоб не займати зайвого місця.

Змінені файли: index.html

Перевірка: venv/bin/python3 -c "import app; print('OK')" — без помилок. HTML-структура валідна: <div class="sticky-controls"> містить .filters і .toolbar, закривається до #statsPanel та .actions-row. Scroll-listener коректно додає/знімає клас .scrolled при прокрутці.

Також оновлено функцію «Розгорнути всі» — тепер вона спочатку відкриває всі сховані секції, а потім розгортає всі картки.

Undo-скасування видалення + виправлення NameError у check_new.py

Проблема/Можливість: 1) У check_new.py на рядку 578 використовувалась змінна checkers, яка не була визначена у цьому scope — при відсутності нових вакансій це призводило б до NameError під час нічного запуску. 2) Кнопка видалення вакансії викликала нативний браузерний confirm() — незручний модальний діалог без можливості скасування після підтвердження. Один невірний клік — і вакансія безповоротно зникала.

Рішення: 1) Виправлено помилку: замінено len(checkers) на просте повідомлення без undefined-змінної. 2) Замінено confirm() на елегантний undo-снекбар у лівому нижньому куті: після кліку «Видалити» вакансія стає напівпрозорою та зникає, а зникне остаточно лише через 5 секунд. Протягом цих 5 секунд з'являється панель «Вакансію видалено — Скасувати» з анімованою кольоровою смужкою-таймером. Натискання «Скасувати» миттєво відновлює вакансію без звернення до сервера. API-запит на видалення надсилається лише після закінчення 5-секундного вікна.

Змінені файли: check_new.py, index.html

Перевірка: python3 -c "import app; print('OK')" та import check_new — без помилок. HTML-структура валідна. Undo-снекбар містить анімований таймер і кнопку «Скасувати», що коректно скасовує видалення.
